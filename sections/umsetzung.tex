\section{Umsetzung des Proof of Concept}
\label{sec:umsetzung}

Technisch werden das 360°-Cockpit sowie das Governance-Cockpit (siehe \autoref{fig:methodology-poc-setup}) mittels einer Python Anwendung mit FastAPI implementiert. Die Nutzung des Python Setups begründet sich damit, dass die CLIs von Anbietern wie Azure und AWS auf Python basieren. Des Weiteren ermöglicht FastAPI mit vergleichbar geringem Aufwand einen RESTful Service zu erstellen. Um die Trennung der beiden Cockpit Komponenten zu betonen, passieren gegenseitige Aufrufe jeweils via HTTP Interface. Zur Persistierung verwenden die Komponenten Project Collection, Evidence Collection, Governance Decision Store und Governance Intent Repository eine MongoDB Document Database. Die Wahl einer Document Database begründet sich mit der Abbildung der Traceability-Beziehung als JSON-Objekte, was eine Handhabung aus Sicht der Implementation vereinfacht. Für den Landing Zone Plattform Configuration Service wird ein statisches IaC Setup mit Terraform verwendet. Die modellierten Traceability-Beziehungen werden hierbei als statischer Inhalt geteilt. Über das Setup hinaus werden weitere Cloud-Ressourcen erzeugt welche in einem funktionalen Setup bereits als gegeben angenommen werden:

\begin{itemize}
    \item Techinsche Benutzer / Service Accounts für die automatisierte wie manuelle Provisionerung von Cloud-Ressourcen
    \item Verrechnungsstruktur als Voraussetzung zur Erzeugung von Subscriptions/Accounts
    \item Initiale Strukturen für Komponenten welche über verschiedenen Landing-Zones hinweg verwendet werden (bspw. Analytics Workspace, Organization Unit, Mangement Groups etc.)
\end{itemize}

Zur Veranschaulichung der technischen Machbarkeit der Traceability-Architektur werden nachfolgend wesentlichen Aspekte der Implementation erläutert. Die Implementierung des \ac{poc} hat sich iherbei auf die Nutzung von Azure und AWS beschränkt umf die Anwendbarkeit auf mehrere cloud-Plattformen zu zeigen

\subsection{UC4 Import der Traceability-Beziehungen}

Im UC4 (siehe \fullref{tab:uc4-define-technical-measures}) werden die verschiedenen Artefakte erzeugt, welche für die Nachvollziehbarkeit der Traceability-Beziehung benötigt werden. Als Grundlage dient das Basismodell für die im \ac{poc} implementiert Variante der Traceability-Architektur \fullref{sec:trace-arch-variant2}, welches in \autoref{fig:data-model-poc} für die technische Implementation erweitert wurde. Als Kontext für die Code-Beispiele wird folgenden Beziehung genommen. Diese Beziehung wird dann in den nachfolgenden Schritten mittels Code-Beispielen dargestellt:

\begin{itemize}
    \item Die Component \glqq Storage \glqq bildet den \ac{lz} Aspekt von Cloud-Ressourcen ab welche zur Datenspeicherung dienen. Hierbei kann es sich Datenbanken, Object-Storage etc. handeln.
    \item Die Regulation \ac{si001} repräsentiert einer der Regelwerke, welche in der \ac{bv} angewendet werden. Für die Storage Component werden hieraus die Abschnitte I2.1 \glqq Vertraulichkeit und Integrität \glqq und I3.2 \glqq Verfügbarkeit \glqq verwendet
    \item Für die Component werden die Compnent-Profiles \glqq standard \glqq und \glqq high \glqq definiert, welche als Baselines zu verstehen sind. High baut hierbei inhaltlich auf Standard auf und verfolgen jeweils 
    \item Die Component-Profiles sind verschiedenen Policies zu geordnet welche mit der durch sie definierten Einschränkungen / Validierung der Absicht der Controls entsprechen sollen. 
\end{itemize}

Die Zuordnung kann häufig durch ein deterministisches Script erreicht werden. Im \ac{poc} wird dieser Tasks der Einfachheit halber mittels KI umgesetzt. Die Auswahl der Controls selbst wird manuell ausgeführt und erhebt keinen Anspruch auf Vollständigkeit

Basierend auf der Baseline der Component-Profiles werden mittels der Beschreibung aus den Regeulationen der ausgewählten Controls I2.1 und I3.2 passende Policies für die unterstützten Cloud-Provider identifiziert und den Profiles pro Plattform zugeordnet (siehe \autoref{fig:component-profile-sample}). Für eine erste Identifkation wird hierbei KI verwendet mit einenm anschliessenden Backing durch manuelle Recherche in den jeweiligen Dokumentationen der Cloud-Provider. Azure bietet mit der zentralisierten Policy-Engine einen direkten Weg die Korrektheit der Policies mittels name oder policy definition id zu überprüfen \footnote{Siehe Policy Definitions in \href{https://portal.azure.com/\#view/Microsoft_Azure_Policy/PolicyMenuBlade.MenuView/\~/Definitions}{Azure Policy Definition Portal}}. AWS hat mit den verschiedenen Services welche eigenen Policy-Engines besitzten einen dezentralisierten Ansatz. Entsprechend muss in der Dokumentation der einzelnen Services die Existenz geprüft werden. Um den unterschiedlichen Ebenen auf denen die Policies angewendet werden gerecht zu werden, wird das Attribut enforcement\_layer auf dem Policy Model eingeführt.
