\section{Umsetzung des Proof of Concept}
\label{sec:umsetzung}

Technisch werden das 360°-Cockpit sowie das Governance-Cockpit (siehe \autoref{fig:methodology-poc-setup}) mittels einer Python Anwendung mit FastAPI implementiert. Die Nutzung des Python Setups begründet sich damit, dass die CLIs von Anbietern wie Azure und AWS auf Python basieren. Des Weiteren ermöglicht FastAPI mit vergleichbar geringem Aufwand einen RESTful Service zu erstellen. Um die Trennung der beiden Cockpit Komponenten zu betonen, passieren gegenseitige Aufrufe jeweils via HTTP Interface. Zur Persistierung verwenden die Komponenten Project Collection, Evidence Collection, Governance Decision Store und Governance Intent Repository eine MongoDB Document Database. Die Wahl einer Document Database begründet sich mit der Abbildung der Traceability-Beziehung als JSON-Objekte, was eine Handhabung aus Sicht der Implementation vereinfacht. Für den Landing Zone Plattform Configuration Service wird ein statisches IaC Setup mit Terraform verwendet. Die modellierten Traceability-Beziehungen werden hierbei als statischer Inhalt geteilt. Über das Setup hinaus werden weitere Cloud-Ressourcen erzeugt welche in einem funktionalen Setup bereits als gegeben angenommen werden:

\begin{itemize}
    \item Techinsche Benutzer / Service Accounts für die automatisierte wie manuelle Provisionerung von Cloud-Ressourcen
    \item Verrechnungsstruktur als Voraussetzung zur Erzeugung von Subscriptions/Accounts
    \item Initiale Strukturen für Komponenten welche über verschiedenen Landing-Zones hinweg verwendet werden (bspw. Analytics Workspace, Organization Unit, Mangement Groups etc.)
\end{itemize}

Zur Veranschaulichung der technischen Machbarkeit der Traceability-Architektur werden nachfolgend wesentlichen Aspekte der Implementation erläutert. Die Implementierung des \ac{poc} hat sich iherbei auf die Nutzung von Azure und AWS beschränkt umf die Anwendbarkeit auf mehrere cloud-Plattformen zu zeigen

\subsection{UC4 Import der Traceability-Beziehungen}

Im UC4 (siehe \fullref{tab:uc4-define-technical-measures}) werden die verschiedenen Artefakte erzeugt, welche für die Nachvollziehbarkeit der Traceability-Beziehung benötigt werden. Als Grundlage dient das Basismodell für die im \ac{poc} implementiert Variante der Traceability-Architektur \fullref{sec:trace-arch-variant2}, welches in \autoref{fig:data-model-poc} für die technische Implementation erweitert wurde. Als Kontext für die Code-Beispiele wird folgenden Beziehung genommen. Diese Beziehung wird dann in den nachfolgenden Schritten mittels Code-Beispielen dargestellt:

\begin{itemize}
    \item Die Component \glqq Storage \glqq bildet den \ac{lz} Aspekt von Cloud-Ressourcen ab welche zur Datenspeicherung dienen. Hierbei kann es sich Datenbanken, Object-Storage etc. handeln.
    \item Die Regulation \ac{si001} repräsentiert einer der Regelwerke, welche in der \ac{bv} angewendet werden. Für die Storage Component werden hieraus die Abschnitte I2.1 \glqq Vertraulichkeit und Integrität \glqq und I3.2 \glqq Verfügbarkeit \glqq verwendet
    \item Für die Component werden die Compnent-Profiles \glqq standard \glqq und \glqq high \glqq definiert, welche als Baselines zu verstehen sind. High baut hierbei inhaltlich auf Standard auf und verfolgen jeweils 
    \item Die Component-Profiles sind verschiedenen Policies zu geordnet welche mit der durch sie definierten Einschränkungen / Validierung der Absicht der Controls entsprechen sollen. 
\end{itemize}

\subsection{UC6 Erstellung der Foundation}

Die Evaluierung und die Provisionierung sind in der \ac{poc} Architektur bewusst in das 360-Cokcpit und das Governancen-Cockpit separiert. Das Governance-Cockpit soll vor allem mit den Elementen arbeiten, welche Teil der Traceability-Architektur sind und möglichst nicht mit Spezifika von Cloud-Providern in Berührung kommen. Das 360-Cockpit hingegen ist für die Ressourcenverwaltung (sprich Foundations) verantwortlich, welche basierend auf den Requests der Projektveranwortlichen erstelle wurden. Entsprechend wird hier auch die Provisionierung verantwortet und Spezifika pro Provider sind bekannt.

Als Grundlage für die Zuweisung von Component-Profiles (und schlussendliche Policies) zu einer Foundation wird eine FoundationDecisionRequestDto verwendet. In dieser formuliert der Projektverantwortliche die Anforderungen welche die Foundation aus Sicher der Governance erfüllen muss.

Den Component Profiles aufsteigend werden die statisch konfigurierten Tresholds mit den Mindestanforderungen der Foundation verglichen. Stimmen diese in allen Werten überein, wurde ein passende Component Profile gefunden. Im Falle eines negativen Matching wird entsprechend ein Leerwert zurückgegeben, welcher oberhalb abgefangen wird

\begin{figure}[H]
\centering
\begin{lstlisting}
def _derive_storage_profile(req: StorageRequirementDto) -> ComponentProfileNameDto:
    for profile in ["standard", "high", "very_high"]:
        limits = STORAGE_PROFILE_LIMITS[profile]

        if req.rpo_minutes is not None and req.rpo_minutes < limits["min_rpo_minutes"]:
            continue

        if req.encryption_at_rest is True and not limits["encryption_required"]:
            continue

        if req.backup_strategy is not None and req.backup_strategy not in limits["allowed_backup_strategies"]:
            continue

        # return the enum member for profile name
        return ComponentProfileNameDto(profile)

    return None
\end{lstlisting}

Basierend auf dem identifizierten Component-Profile der datenbezogenen Anforderungen wird nachträglich eine Korrektur vorgenommen. Diese soll sicherstellen, dass die Sensitivität der bearbeitenden Daten berücksichtigt wird, da diese einen Einfluss auf alle Components hat. Für den \ac{poc} wird anschliessend die Governance-Domain identifiziert, welche den Anforderungen der Foundation genau oder mindestens entsprechen. Dies geschieht mittels eines Scoring pro Governance-Domain / \ac{lz}. Bei diesem Scoring wird gemessen wie gross die Abstände zwischen den für die Foundation identifizierten Component-Profiles zu jenen der vordefinierten Governance-Domain sind. Governance-Domain mit zu tiefen Component-Profile werden bereits vorher verworfen.

\begin{figure}[H]
\centering
\begin{lstlisting}

req_network = effective_network

....

lzs: list[LandingZone] = ...

# select eligible landing zones where each component profile >= requested profile
candidates = []
for lz in lzs:
  # component_profiles may use enum keys/values; normalize to str->str mapping
  cp = lz.component_profiles or {}
  cp_str = {}
  for k, v in cp.items():
    key = k.value
    val = v.value
    cp_str[str(key)] = str(val)

  lz_compute = cp_str.get("compute", "standard")
  lz_network = cp_str.get("network", "standard")
  lz_storage = cp_str.get("storage", "standard")

  try:
    ok = (
        rank.get(lz_compute, 0) >= rank.get(req_compute, 0)
        and rank.get(lz_network, 0) >= rank.get(req_network, 0)
        and rank.get(lz_storage, 0) >= rank.get(req_storage, 0)
    )
  except Exception:
    ok = False

  if ok:
    # score is closeness (lower is better)
    score = (
        (rank[lz_compute] - rank[req_compute])
        + (rank[lz_network] - rank[req_network])
        + (rank[lz_storage] - rank[req_storage])
    )
    candidates.append((score, lz))

if not candidates:
    logger.warning("No landing zone available that meets or exceeds requested component profiles")
    return None

# choose best candidate (minimal score)
candidates.sort(key=lambda x: (x[0], x[1].id))
chosen = candidates[0][1]
...
