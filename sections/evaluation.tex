\textbf{Performance}
Der wichtigste Aspekt für die Performance ist die Prüfung der Compliance. Bei dieser wird für jede Policy, welcher einem Component-Profile zugeordnet wird eine Abfrage gegenüber dem Cloud Provider API getätigt. Daraus entsteht eine hohe Netzwerklatenz da eine Batch-Abfrage nicht als gegeben angenommen werden kann.

\textbf{Security}
Security Ziele können implizit durch die Definition von Controls, Regulations und Policies abgebildet und erforced werden.

\textbf{Portability}
Die Controls und Regulations selbst sind nicht plattformspezifisch formuliert und können deshalb auf weitere Cloud-Plattformen übertragen werden. Eine Verknüpfung von plattformspezifischen Policies ist zum Enforcment erforderlich.

\textbf{Architectural Sustainability}
Die Abbildung mit Controls lässt es bspw zu bei Änderungen einer Regulation wie \ac{si001} weitere Controls abzubilden. Voraussetzug für die Erweiterbarkeit ist jedoch dass weitere Regulations jeweils eine ähnliche control-basierte Strktur aufweisen, sprich sich in Abschnitte an Security Statements unterteilen lassen.

\textbf{Auditability}
Für eine Zuweisung einer Foundation lässt sich via die Metainformationen der Controls und Regulation ein Kontext ableiten. Aus diesen Kontext geht hervor wer ein Objekt der Traceability-Relationsship definiert oder zu welchem Zeitpunkt geändert hat.

\textbf{Automatibility}
Das Deployment von Traceability-Beziehungen und die Nachvollziehbarkeit von Beziehungen ist durch die explizite Abbildung zu einem hohen gerade automatisierbar. Die Abbildungen von Policies der einzelnen Plattformen zu den Controls kann ebenfalls automatisiert via Ki erfolgen.

\textbf{Maintainabiliy}
Regulations wie Controls sind versionierbar. Entsprechend ist eine Erweiterung bzw Update von Objekten der Traceability-Beziehung möglich ohne bestehenden Deployment zu beeinflussen.
