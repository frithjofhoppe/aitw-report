Um ein Verständnis über die Idee der \ac{lz} (wie in der Einleitung dargelegt) zu erhalten werden die relevanten Cloud-Provider betrachtet. Die Unternehmen Amazon mit der Plattform AWS, Google mit GCP und Microsoft mit Azure bilden hierbei nach \cite{saraswat_cloud_2020} die drei marktstärksten Anbieter von umfassenden Cloud-Services. Zu den genannten Anbietern werden zusätzlich noch IBM, Alibaba und Oracle wegen Verträgen im Rahmen des Vorhabens "Public Clouds Bund" der WTO-20007 nach \cite{bk_public_2025} ebenfalls einbezogen, da diese vom \ac{bit} verwendet werden. Diese Anbieter werden im Folgendem weiter als Cloud-Provider verstanden.

In einer initialen systematischen Literatur-Recherche wird ermittelt wie etabliert der Begriff \ac{lz} bereits in wissenschaftlichen Arbeiten und der Industrie ist. Ein Report dazu zeigt \cite{elicit_what_2025} (siehe \autoref{appendix:elicit-report-landingzone}), dass bis auf eine Quelle die Terminologie \ac{lz} nicht explizit verwendet wird. Viel mehr werden auf grundsätzliche architektonische Konzepte hingewiesen, welche mit der Idee der \ac{lz} zusammenhängen. Quellen welche den Begriff verwenden sind bis auf wenige Ausnahmen \parencite{laheri_designing_2025} Dokumente der Cloud-Provider selbst oder Fachliteratur welche wie bei \cite{mulder_multi-cloud_2020} indirekt auf die konkrete Implementierung von Cloud-Providern verweist. In ersten Architektur-Blogs von AWS \parencite{orban_four_2017} wird der Begriff als Implementation einer der sogenannten \textit{6R} verwendet \parencite{orban_6_2016}. Bei diesen \textit{R} handelt es sich um die von \cite{gartner_gartner_2013} ursprünglich fünf erkannten Ansätze von Organisationen in die Cloud zu migrieren. Basierend auf diesen Ansätzen nennt \cite{orban_four_2017}, dass als Teil der Migration das bestehende Applikations-Portfolio nach Befürfnissen gruppiert werden soll, um passenden Operation- und Application-Landing-Zones zu identifizieren. Die \ac{lz} definieren high-level Architekturpatterns und Capabilities \parencite{orban_four_2017}

Im Rahmen des IP5 Projektes (siehe \fullref{appendix:ip5}) wurde sich von den Autoren dieser Arbeit bereits mit der Thematik der plattformübergreifenden Governance mittels Policies  mit Fokus auf Azure und AWS auseinandergesetzt. Mit einem PoC wurde gezeigt, dass eine agnostische Implementation grundsätzlich möglich ist.
