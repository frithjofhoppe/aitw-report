Das Ziel dieser Arbeit ist es die relevanten Komponenten sowie eine beispielhafte Struktur einer \ac{lz} aufzuzeigen. Die Ableitung von einer groben \ac{lz}-Architektur zeigt, dass eine Orientierung am Business-Need ein wichtiger Faktor ist, um eine nutzbare Architektur zu erhalten. Dafür wird in dieser Arbeit innerhalb des Konzepts einer \ac{ea} eine Capability-Map erstellt. Mit einem Teil der Capabilities wird anschliessend eine Traceability-Architektur als mögliche Ausprägung einer \ac{lz}-Architektur formuliert und evaluiert. Die Evaluation zeigt, dass basierend auf Kriterien wie Architekturprinzipien, Qualitäts-Attribute sowie Trade-Off Dimensionen sich die Variante der Component-centric Traceability-Architektur Variante für den Use-Case des \ac{bit} eignet.

\subsection{Beantwortung der Forschungsfragen}
\label{sec:answer-research-question}

% \begin{itemize}
%     \item Explizite Beantwortung der Forschungsfragen auf Basis der erzielten Ergebnisse
%     \item Einordnung der wichtigsten Architekturprinzipien für strukturierte Landing-Zone-Architekturen
% \end{itemize}

Als Zusammenfassung der Evaluation und Diskussion werden die relevanten Inhalte für die anfänglich definierten Forschungsfragen formuliert.

\textit{Welche Komponenten machen eine Landingzone für eine Hybrid-Multi-Cloud-Umgebung aus?}

Die \ac{lz} wird in dieser Arbeit als technisches Gerüst betrachtet, welches die verschiedenen Bedürfnisse des Businesses hinsichtlich einer einheitlichen Governance implementiert. Die Komponenten und deren Scope in der \ac{lz} werden somit durch das Business definiert. Diese Arbeit verwendet dafür eine Capability-Map (siehe \fullref{fig:reduced-capabiliy-map}) und definiert folgende Komponenten: \flqq Identity and Access Governance\frqq, \flqq Security Governance\frqq, \flqq Policy and Compliance Lifecycle Governance\frqq, \flqq Network and Connectivity\frqq, \flqq Data and Secret Governance\frqq, \flqq Observability and operational Insight Governance\frqq\ sowie \flqq Cost Governance\frqq. Basierend auf diesen Komponenten lässt sich eine \ac{lz}-Architektur und somit auch eine \ac{lz} modellieren.

\textit{Wie sieht eine geeignete technische Architektur einer Landingzone für eine Hybrid-Multi-Cloud-Umgebung aus und was muss sie erfüllen?}

Eine \ac{lz}-Architektur auf \ac{ea}-Ebene muss für die multi-Cloud Umgebung gewisse Architekturprinzipien erfüllen. Diese fokussieren sich vom Charakter her auf eine Plattformunabhängigkeit und vor allem auf eine Konformität mit der \ac{ea} der Organisation für welche die \ac{lz}-Architektur entworfen wird. In dieser Arbeit werden die Prinzipien für den Use-Case des \ac{bit} in die folgenden Kategorien unterteilt: \textit{Governance} (mit Compliance with Law and Policies, Data Governance), \textit{Abstraction and Plattform Independency} (mit Cloud Native by Default, Architectural Abstraction Principal, Product Neutrality), \textit{Structural Architecture} (mit Serviceorientation, Modular Decomposition Principle, Reference Model Principle, Standardisation Automation) und \textit{Security-Architecture} (Security by Design, Defence-in-Depth, Security Reconfigurability Principle) unterteilt. Basierend auf diesen Prinzipen wird die Grundarchitektur für eine \ac{lz} definiert (siehe Modellierung mit ArchiMate in \autoref{fig:landing-zone-architecture}). In dieser Architektur wird der \ac{lz}-Service als Technology Services modelliert, welche thematisch eine der Capabilities zusammenfasst. Die Ausprägungen hiervon sind die \ac{lz}-Components wie in der ersten Forschungsfrage beschrieben. \ac{lz}-Components können dabei ebenfalls als shared Component vorkommen. Die Implementation auf den Plattformen wird durch die jeweiligen Plattform Components dargestellt.

\subsection{Beitrag der Arbeit}

Diese Arbeit schlägt eine \ac{lz}-Architektur auf \ac{ea}-Ebene vor und zeigt mit einer Traceability-Architecture auf, wie eine Teil-Implementation einer \ac{lz} aussehen könnten. Für die allgemeine Ausarbeitung einer \ac{lz} propagiert diese Arbeit einen Business-driven Ansatz, bei dem mittels einer Capability-Map einzelne \ac{lz}-Komponenten abgeleitet werden.

\subsection{Ausblick}

% \begin{itemize}
%     \item Weiterführende Evaluation weiterer Capabilities
%     \item Anwendung des Ansatzes in realen Transformationsprojekten
%     \item Untersuchung organisatorischer und wirtschaftlicher Auswirkungen

Diese Arbeit setzt sich mit der Architektur einer \ac{lz} auf der Ebene der \ac{ea} auseinander. Für eine direkte Anwendung ist jedoch eine detaillierte Ausarbeitung in Form von Lösungsarchitekturen und entsprechenden Implementationen nötig. Analog zu der Traceability-Architektur könnte man somit mit den verbleibenden Capabilities neue Artefakte ableiten zum Implementieren. Insbesondere durch die Anwendung in einer Unternehmensumgebung liessen sich somit Erkenntnisse sammeln, wie und ob der vorgeschlagene Ansatz in der Realität funktioniert. Des Weiteren kann die Auswahl von geeigneten Architekturprinzipien für die \ac{lz} durch einen eher empirischen Ansatz gehärtet werden.
