\section{Umsetzung des Proof of Concept}
\label{sec:umsetzung}

Technisch werden das 360°-Cockpit sowie das Governance-Cockpit (siehe \autoref{fig:methodology-poc-setup}) mittels einer Python Anwendung mit FastAPI implementiert. Die Nutzung des Python Setups begründet sich damit, dass die CLIs von Anbietern wie Azure und AWS auf Python basieren. Des Weiteren ermöglicht FastAPI mit vergleichbar geringem Aufwand einen RESTful Service zu erstellen. Um die Trennung der beiden Komponenten 360°-Cockpit und Governance-Cockpit zu betonen, passieren gegenseitige Aufrufe jeweils via HTTP Interface. Zur Persistierung verwenden die Komponenten Project Collection, Evidence Collection, Governance Decision Store und Governance Intent Repository eine MongoDB Document Database. Die Wahl einer Document Database begründet sich mit der Abbildung der Traceability-Beziehung als JSON-Objekte, was eine Handhabung aus Sicht der Implementation vereinfacht. Für den Landing Zone Plattform Configuration Service wird ein statisches \ac{iac} Setup mit Terraform verwendet. Für AWS stellt dies eine Organization Unit mit zugewiesenen Service Control Policies da. Für Azure wird eine Management Group mit Policy Assignment auf dieser verwendet.

Die modellierten Traceability-Beziehungen werden hierbei als statischer Inhalt geteilt. Über das Setup hinaus werden weitere Cloud-Ressourcen erzeugt welche in einem funktionalen Setup bereits als gegeben angenommen werden:

\begin{itemize}
    \item Technische Benutzer / Service Accounts für die automatisierte wie manuelle Provisionierung von Cloud-Ressourcen
    \item Verrechnungsstruktur als Voraussetzung zur Erzeugung von Subscriptions/Accounts
    \item Initiale Strukturen für Komponenten welche über verschiedene Landing-Zones hinweg verwendet werden (z.B. Analytics Workspace, Organization Unit, Management Groups, etc.)
\end{itemize}

Zur Veranschaulichung der technischen Machbarkeit der Traceability-Architektur werden nachfolgend wesentliche Aspekte der Implementation erläutert. Die Implementierung des \ac{poc} hat sich hierbei auf die Nutzung von Azure und AWS beschränkt u die Anwendbarkeit auf mehrere Cloud-Plattformen zu zeigen.

Im UC4 (siehe \hyperlink{usecase-uc4}{Beschrieb UC4}) werden die verschiedenen Artefakte erzeugt, welche für die Nachvollziehbarkeit der Traceability-Beziehung benötigt werden. Als Grundlage dient das Basismodell für die im \ac{poc} implementierte Variante der Traceability-Architektur \fullref{sec:trace-arch-variant2}, welches in \autoref{fig:data-model-poc} für die technische Implementation erweitert wurde. Als Kontext für die Code-Beispiele werden folgende Beziehungen genommen, welche dann in den nachfolgenden Schritten mittels Code-Beispielen dargestellt werden:

\begin{itemize}
    \item Die \texttt{Component} \flqq Storage\frqq\ bildet den \ac{lz} Aspekt von Cloud-Ressourcen ab welche zur Datenspeicherung dienen. Hierbei kann es sich um Datenbanken, Object-Storage, usw. handeln.
    \item Die \texttt{Regulation} \ac{si001} repräsentiert eines der Regelwerke, welche in der \ac{bv} angewendet werden. Für die Storage Component werden hieraus die Abschnitte I2.1 \flqq Vertraulichkeit und Integrität\frqq\ und I3.2 \flqq Verfügbarkeit\frqq\ verwendet.

    \item Den \texttt{Component\_Profile}s sind verschiedene \texttt{Policy}s zugeordnet, welche durch die definierten Einschränkungen / Validierung der Absicht der Controls entsprechen sollen.
\end{itemize}

\subsection{Vorbedingung von UC6: Erstellung der Foundation}

Die Evaluierung und die Provisionierung (siehe Vorbedingung \hyperlink{usecase-uc6}{Beschrieb UC6}) sind in der \ac{poc} Architektur bewusst separiert worden in das 360°-Cockpit und das Governance-Cockpit. Das Governance-Cockpit soll vor allem mit den Elementen arbeiten, welche Teil der Traceability-Architektur sind und möglichst nicht mit Spezifika von Cloud-Providern in Berührung kommen. Das 360°-Cockpit hingegen ist für die Ressourcenverwaltung (z.B. innerhalb der Foundations) verantwortlich, welche basierend auf den Requests der Projektveranwortlichen erstellt wurden. Entsprechend wird hier auch die Provisionierung verantwortet und Spezifika pro Provider sind bekannt.

\begin{minted}[breaklines]{python}
def _derive_storage_profile(req: StorageRequirementDto) -> ComponentProfileNameDto:
    for profile in ["standard", "high", "very_high"]:
        limits = STORAGE_PROFILE_LIMITS[profile]

        if req.rpo_minutes is not None and req.rpo_minutes < limits["min_rpo_minutes"]:
            continue

        if req.encryption_at_rest is True and not limits["encryption_required"]:
            continue

        if req.backup_strategy is not None and req.backup_strategy not in limits["allowed_backup_strategies"]:
            continue

        # return the enum member for profile name
        return ComponentProfileNameDto(profile)

    return None

Basierend auf dem identifizierten Component-Profile der datenbezogenen Anforderungen wird nachträglich eine Korrektur vorgenommen. Diese soll sicherstellen, dass die Sensitivität der bearbeitenden Daten berücksichtigt wird, da diese einen Einfluss auf alle Components hat. Für den \ac{poc} wird anschliessend die Governance-Domain identifiziert, welche den Anforderungen der Foundation genau oder mindestens entsprechen. Dies geschieht mittels eines Scorings pro Governance-Domain bzw. \ac{lz}. Bei diesem Scoring wird gemessen wie gross die Abstände sind zwischen den für die Foundation identifizierten Component-Profiles und jenen der vordefinierten Governance-Domain. Governance-Domains mit zu tiefen Component-Profiles werden bereits vorher verworfen.

\subsection{UC6: Prüfung der Compliance}

Der Stand der Compliance einer Foundation kann geprüft werden (siehe \hyperlink{usecase-uc6}{Beschrieb UC6}), sobald die Provisionierung durch das 360°-Cockpit stattgefunden hat. Im Rahmen vom \ac{poc} wurde lediglich die Compliance Prüfung auf Azure vollständig umgesetzt. Ausgangspunkt der Prüfung ist die Governance Decision, aus welcher die Component-Profiles abgelesen werden. Basierend auf den Component-Profiles werden alle Policies zu einer jeweiligen Control-Id aggregiert. Die Auswertung mit Fokus auf die Control begründet sich damit, dass schlussendlich mit allen evaluierten Controls einer Regulation der Compliance Stand der Regulation abgeleitet werden kann. Der Compliance Stand einer Control zum Zeitpunkt der Durchführung wird dann als Evidence im Governance Decision Store abgespeichert wie in \autoref{fig:evidence-generation-flow} ersichtlich ist. Der Prozess den Compliance-State zu prüfen sollte idealerweise in regelmässigen Abständen durchgeführt werden, damit schlussendlich Akteure wie der Compliance Officer ein möglichst aktuelles Lagebild im Governance-Cockpit haben, welches auf den Evidences aufbaut.

Die Evidence View basiert auf dem \texttt{GET /project/\{project\_id\}/evidence} Endpoint. Dieser Endpoint sucht für alle Foundations welche dem Projekt mit der id \texttt{project\_id} zugeordnet sind den letzten verfügbaren Compliance State pro Control. Diese Auswertung findet wenn möglich direkt in MongoDB statt zur Reduktion der Datenmenge welche zur Laufzeit geladen wird (in \autoref{fig:query-lates-foundation-compliance-state} gezeigt).
