\section{Landing Zone Architektur}
\label{sec:landingzone-architecture}

Eine \ac{lz} stellt das konzeptionelle Grundgerüst dar in welchem der Workload für ein Projekt existiert (siehe \fullref{sec:methodik-landingzones}). Der Begriff Foundation ergibt sich aus den Use-Cases, welche den Scope der Arbeit setzen und sich auf das \ac{bit} beziehen.

Die \ac{lz} bildet trotz der schlussendlichen technischen Implementation bei distanzierter Betrachtung verschiedene Themengebiete ab. So ist der Nutzen der \ac{lz} die verschiedenen Gebiete so zu modellieren, dass sie dem Interesse des Unternehmens / der Organisation entsprechen. Hiermit ist gemeint, dass bspw. technische, strukturelle wie regulatorischen Vorgaben der Organisation in Bezug auf den Workload angewendet werden. Die Anwendung passiert hierbei implizit oder explizit durch die Platzierung des Workloads in der \ac{lz} bzw Foundation. Die Themengebiete, welche relevant für die \ac{lz} sind, lassen sich aus sich der \ac{ea} auf technische Capabilities zurückführen (siehe \fullref{sec:methodik-capability}). Die technischen Capabilities stellen somit das technischen \textit{Was} da, welche eine Organisation in Bezug auf die Workloads können möchte. Die \ac{lz} auf der Gegenseite stellt das \textit{Wie} bzw. einen Teil der Implementation da. Diese verschiedenen Themengebiete, in welchem die technischen Capabilities resultieren, werden nachfolgenden als \ac{lz}-Component bzw Component bezeichnet. Bei einer vollständigen Implementierung sollten somit theoretisch alle Capabilities in mindestens einer \ac{lz}-Component vertreten sein. Repräsentativ für die diversen identifizierten high-level / Level 1 Capabilities in \autoref{fig:reduced-capabiliy-map} wird für die Arbeit passend für den Use-Case (siehe \autoref{fig:traceability-use-cases}) eine Auswahl vorgenommen. Die Use-Cases (siehe \autoref{fig:traceability-use-cases}) basierend auf der Fähigkeit der Traceability welche im \autoref{sec:traceability-architecture} eingehender beschrieben wird.

Diese Capability der Traceability hängt in der Capability Map an den Level 1 Capabilities Security Governance, Policy and Compliance Lifecycle Governance, Audit and Evidence Governance, Configuration and Standards Governance. Mittels der Traceability wird exemplarische eine allgemeine \ac{lz} Architektur in \autoref{fig:landing-zone-architecture} aufgebaut. Zentraler Angelpunkt in dieser Architektur sind die \ac{lz} Services. Diese Services bilden generelle Blöcke an Ressourcen da. Denkbar hier sind beispielsweise Storage-Services, Identity-Services etc. welche sich an den Level 1/2 Capabilities orientieren. Aufbauend auf den \ac{lz} Services werden die \ac{lz} Components und die Shared Solution Components abgeleitet. Shared Solution sind zentrale Components welchen über nicht an den Lifecycle von Foundations gebunden sind. Beispiele können hier \ac{iam}-, Audit- oder Site-to-Site-Serivces sein. \ac{lz}-Components hingegen orientieren sich an den Lifecycle einer Foundation. Sie orientieren sich ebenfalls den Level 1/2 Capabilities sind aber nur spezifische für den Scope einer Foundation gedacht. Basierend auf dem \ac{iam} Beispiel kann als Gegenstück für einen Shared Solution \ac{iam} Component eine foundation-spezifische Component existieren, welche die Integration sicherstellt. Die Plattform Component am Ende der Struktut stellt als Technology Node eine plattform-spezifische Komponenten da, mit welchem die Technology-Services auf den Cloud-Plattformenen umgesetzt werden. Für die Anwendung des \ac{lz} Konstrukt wird ein \ac{lz} Blueprint verwendet. Diese Blueprints stellen Artefakte da welche auf den \ac{lz} Components aufbauen und im Sinne von \ac{iac} provisioniert werden können.

\subsection{Traceability Architektur}
\label{sec:traceability-architecture}

Die Traceability-Architektur stellt eine mögliche Implementation da, wie Capabilities realisierte werden können. Die Traceablity-Architektur ist somit eine Teilarchitektur im grösseren \ac{ea} Kontext und beeinflusst die \ac{lz}-Architektur wie in \autoref{fig:traceability-architecure-artifacts} ersichtlich ist. Traceability wird hierbei als Fähigkeit verstanden den Weg von Anforderungen aus einer Regulation wie \ac{si001} hin bis zu einer Cloud-Ressource nachvollziehen zu können. Dies umfasst sowohl die Aussage, dass ein Objekt wie eine Control aus der Regulation für die Cloud-Ressource relevant ist, als auch den Compliance-Status. Der Compliance-Status sagt wiederum aus, ob die Anforderungen aus der Control gegenüber der Cloud-Ressource erfüllt sind. Summiert man dies auf alle Controls für eine Foundation auf, kann somit mittels der Traceability-Architektur eine Aussage gemacht werden über die Compliance von Foundations oder Projekten als Ganzes. Die Traceability soll somit folgenden Probleme lösen:

\begin{itemize}
    \item Wie weist man die relevanten Controls aus einer Regulation zu einer Foundation zu? Der Workload von Projekten in der Cloud muss Vorgaben einhalten, entsprechend muss ein Bezug zwischen Anforderung und Workload hergestellt werden können.
    \item Wie kann ein Audit für eine Foundation erstellt werden? Eine Organisation wie das \ac{bit} muss eine Aussage machen können ob der Workload von Projekten konform läuft.
\end{itemize}

Die Kernidee der Traceability-Architektur ist es diese Probleme mittels einer Traceability-Beziehung / -Relation zu lösen. Als Relation wird jene zwischen den Controls aus einer Regulation zu den Cloud-Ressourcen bezeichnet, welche in einer Foundation laufen. Diese Beziehung soll explizit modelliert werden, um transparent nachvollziehen zu können welche Controls angewendet werden. Diese Beziehung kann dann verwendet werden um unter anderem Audits durchführen zu können.

Die zentralen Artefakte in der Architektur bauen auf einem Basismodell auf, welche als Grundlage für die Capabilities erstellt wurde. Eine Regulation bildet ein Regelwerk wie z.B. \ac{si001} ab, welches die verschiedenen Controls enthält. Eine Control ist je eine Anforderung welche eingehalten werden muss damit die Control als eingehalten gilt. Eine Control kann durch eine oder mehrere Policies implementiert werden, welche die konkrete Umsetzung der Control darstellen. Diese Policies können entweder selbst definiert oder von den Cloud-Providern bereitgestellt werden.

Die formulierte Idee der Traceability-Architektur auf \ac{ea} Ebene kann unterschiedlich in der Ebene der Lösungsarchitektur umgesetzt werden. Um verschiedene Aspekte der Traceability besser beschreiben zu können wird deshalb im Rahmen eines \ac{poc} ein Beispiel für eine Lösungsarchitektur gegebenen (siehe \fullref{sec:poc-methodology}). Diese Architektur wird nachfolgend sowie in den verschiedenen Varianten / Ansätzen der Lösungsarchitektur verwendet.

Für eine Prüfung der Compliance von einem Workload ist initial ein Deployment einer Foundation erforderlich (siehe \hyperlink{usecase-uc6}{UC-6}). Dazu werden die Anforderungen an die Governance der Foundation durch den Projektverantwortlichen generell formuliert (siehe \autoref{fig:deploy-sequence-base-structure}). Basierend auf diesen Anforderungen wird pro Komponente in der Lösungsarchitektur eine Placement-Decision getroffen. Als Placement wird hierbei eine Beurteilung der Compliance verstanden. Diese Beurteilung enthält unter anderen welche Teile einer Regulation relevant sind. Schlussendlich wird die Decision persistiert und basierend auf ihr eine Foundation provisioniert, welche den Workload enthält.

\subsection{Variante 1: Control-centric Bundling}
\label{sec:trace-arch-variant1}

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{graphics/ip6-ideation-Traceability-Architecture-control-centric.drawio.pdf}
    \caption{Control-centric Traceability Architektur Variante}
    \label{fig:traceability-architecture-control-centric}
\end{figure}

\textbf{Ansatz}: Für jede Control in der jeweiligen Regulation wird ein Mapping zu einem Plattform-Bundle gemacht.

\textbf{Fokus}: Traceability, da die Controls direkt mit den Bundles verknüpft sind.

\textbf{Artefakte}: versionierte Bundles z.B. in Git, Governance Decisions und Evidence als Datenobjekte

\textbf{Funktionsweise}: In dieser Architektur-Variante werden die Controls direkt mit den Plattform-Bundles verknüpft. Das bedeutet, dass für jede Control in der Regulation ein spezifisches Mapping zu einem oder mehreren Bundles besteht, die die entsprechenden Policies enthalten. Beim Deployment einer \ac{lz} wählt das Governance Cockpit die benötigten Controls aus, basierend auf den Anforderungen der \ac{lz} (z.B. Schutzbedarf, Compliance-Anforderungen, etc.).

\textbf{Verantwortlichkeiten}: Der Projektverantwortliche definiert die Anforderungen für die jeweilige Foundation. Der Compliance Officer ist zuständig für die Regulations und Controls. Er prüft zudem das Mapping der Bundles durch den Cloud-Experten sowie die Zuweisung der Bundles zu den Foundations.

\textbf{Maintenance}: Da jedes Control direkt mit einem Bundle verknüpft ist, erfordert die Pflege und Aktualisierung der Controls eine sorgfältige Verwaltung der Bundles. Änderungen an einem Control können Auswirkungen auf mehrere Bundles haben, was eine Versionierung und koordinierte Aktualisierung erfordert.

\textbf{Adaptability}: Neue Controls können durch die Erstellung und Verknüpfung neuer Bundles hinzugefügt werden. Dies setzt jedoch einen soliden Lifecycle-Prozess voraus, sowohl auf Seite des \ac{iac} sowohl auch des Cloud-Providers. Die Komplexität der Auswahl der richtigen Bundles für eine \ac{lz} nimmt zu, wenn die Anzahl der Controls und Bundles wächst.

\textbf{Abgrenzung zu anderen Varianten}: Im Vergleich zu Variante 2, welche die Controls in Component-zentrierte Profiles bündelt, ist diese Variante stärker auf die direkte Verknüpfung von Controls mit Bundles fokussiert, was eine klarere Rückverfolgbarkeit ermöglicht. Variante 3 hingegen bündelt die Controls auf einer höheren Ebene, nämlich auf der Ebene von \ac{lz}-Kategorien, was zu einer stärkeren Abstraktion führt und die Rückverfolgbarkeit eher kompliziert.

\textbf{Tracebility-Strength}: Die Rückverfolgbarkeit zu einzelnen Controls ist in dieser Variante am stärksten, da die Controls direkt mit den Bundles verknüpft sind. Es ist einfach zu verfolgen, welche Bundles welche Controls implementieren, was eine klare Rückverfolgung ermöglicht.

\textbf{Laufzeitkomplexität}: Deploy Foundation: O(n), Verify Compliance: O(n)

Beim Deployment einer neuen \ac{lz} wird in Variante 1 zur Ermittlung der Governance Domain im Intent Repository nach verfügbaren Control-Bundles geschaut, welche die relevanten Controls enthalten. Es wird also direkt nach den Controls geschaut, da diese direkt mit den Bundles verknüpft sind. Es ist somit einfach zu verfolgen, welche Bundles welche Controls implementieren, was eine klare Rückverfolgung ermöglicht.

\textbf{Pros}
\begin{itemize}
    \item Rückverfolgbarkeit ist trivial: Das Ergebnis des Enforcements wird direkt einer Control-ID zugeordnet.
    \item Ideal für Audits (\flqq Zeig mir alle implementierten Controls\frqq).
\end{itemize}

\textbf{Cons}
\begin{itemize}
    \item Viele kleine Artefakte → erfordern Automatisierung und eine Build-Pipeline
    \item Einige Provider bevorzugen grössere Sets
\end{itemize}

\textbf{Fazit}: Diese Variante ist diejenige welche am besten die Traceability Anforderungen erfüllt, aber auch die aufwändigste in der Umsetzung ist. Es könnte als \flqq Goldstandard\frqq\ für die Traceability Architektur angesehen werden, aber es muss evaluiert werden ob der Aufwand gerechtfertigt ist im Vergleich zu den anderen Varianten.

\subsection{Variante 2: LZ component-centric Bundling}
\label{sec:trace-arch-variant2}

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{graphics/ip6-ideation-Traceability-Architecture-component-centric.drawio.pdf}
    \caption{Component-centric Traceability Architektur Variante}
    \label{fig:traceability-architecture-component-centric}
\end{figure}

\textbf{Ansatz}: Die Controls werden nach \ac{lz}-Komponenten in Component-Profiles gebündelt.

\textbf{Fokus}: Engineering, Verteilung an Expertengruppen/Teams, aber mit Rückverfolgbarkeit

\textbf{Artefakte}: versionierte Component-Profiles z.B. in Git, Governance Decisions und Evidence als Datenobjekte

\textbf{Funktionsweise}: Die Controls haben mind. eine Beziehung zu mind. einer \ac{lz}-Komponente, welche die Control implementiert. Für jede \ac{lz}-Komponente existieren verschiedene Profiles, diese Profiles bauen aufeinander auf, heisst die Nächsthöhere enthält Vorherige auch. Die Component Profiles repräsentieren die jeweiligen Abstufungen in den Regulations, z.B. bei \ac{si001} könnte die Sicherheitsstufe \flqq hoch\frqq\ das Component Profile \flqq storage high\frqq\ enthalten. Bei mehreren angewendeten Regulations gewinnt jeweils das mit den höheren Anforderungen. Ein Profile kann jeweils mehrere Controls implementieren und bildet das Mapping zwischen Controls und konkreten Policies auf der jeweiligen Zielplattform. Policies können sowohl selbst definiert als auch vom Provider bereitgestellt sein. Der Cloud-Experte ist zuständig für das Mapping der Component-Profiles auf die Controls.

\textbf{Verantwortlichkeiten}: Der Projektverantwortliche definiert die Anforderungen für die jeweilige Foundation. Der Compliance Officer ist zuständig für die Regulations und Controls. Er prüft zudem das Mapping der Component-Profiles durch den Cloud-Experten sowie die Zuweisung der Component-Profiles zu den Foundations.

\textbf{Maintenance}: In dieser Architektur-Variante werden die Controls in Component Profiles gebündelt, folgend erfordern Änderungen an einer Control nur die Aktualisierung des entsprechenden Profiles, welche versionierbar sind. Durch die Versionierbarkeit kann z.B. ein Compliance Officer neue Versionen der Component-Profiles erstellen, ohne dass die bestehenden Foundations direkt betroffen sind. Sobald eine neue Version eines Component-Profils freigegeben ist, können die entsprechenden Foundations und die übergreifende Landing-Zone-Components per Provisionierung migriert werden.

\textbf{Adaptability}: Neue Controls können durch die Erstellung und Verknüpfung neuer Versionen von Profiles hinzugefügt werden. Dies ermöglicht eine flexible Anpassung an neue Anforderungen, da die Profiles auf Komponentenebene organisiert sind und Änderungen an einer Control nicht zwangsläufig Auswirkungen auf andere Controls oder Bundles haben.

\textbf{Abgrenzung zu anderen Varianten}: Ähnlich wie in Variante 1 werden hier Controls \flqq gebundelt\frqq\ aber anstatt in Control-zentrierte Bundles, werden sie in Component-zentrierte Profiles gebündelt. Die Profiles sind somit eher nach technischen Komponenten organisiert, welche von Expertengruppen/Teams implementiert werden, anstatt direkt nach Controls. Variante 3 im Gegenzug bündelt die Controls auf einer noch höheren Ebene, nämlich auf der Ebene von \ac{lz}-Kategorien, was zu einer stärkeren Abstraktion führt.

\textbf{Tracebility-Strength}: Die Traceability ist durch die Verknüpfung der Controls mit den Component Profiles gegeben, da jedes Profile die spezifischen Controls enthält, die es implementiert. Allerdings erfordert die Rückverfolgbarkeit zu einzelnen Controls einen zusätzlichen Suchschritt, da die Controls in den Profiles gebündelt sind. Es ist notwendig, ein Manifest zu führen, das auflistet, welche Controls in welchen Component-Profilen enthalten sind, um eine klare Rückverfolgung zu ermöglichen.

\textbf{Pros}
\begin{itemize}

    \item Weniger Bundles, einfachere Einführung
\end{itemize}

\textbf{Cons}

\end{itemize}

\textbf{Fazit}: Diese Variante bietet einen guten Kompromiss zwischen Rückverfolgbarkeit und Praktikabilität, da sie die Implementierungspraxis der Teams berücksichtigt, aber dennoch eine gewisse Rückverfolgbarkeit ermöglicht. Es könnte als \flqq Silberstandard\frqq\ für die Traceability Architektur angesehen werden, da es eine praktikable Lösung darstellt, ohne den Aufwand der Control-zentrierten Variante zu erfordern.

\subsection{Variante 3: LZ category centric Bundling}
\label{sec:trace-arch-variant3}

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{graphics/ip6-ideation-Traceability-Architecture-category-centric.drawio.pdf}
    \caption{LZ Category centric Traceability Architektur Variante}
    \label{fig:traceability-architecture-category-centric}
\end{figure}

\textbf{Ansatz}: Controls werden nach \ac{lz}-Categories gebündelt, welche die Rules für alle Komponenten enthalten. Diese \ac{lz}-Categories umfassen die entsprechenden Policies. Beim deployen einer \ac{lz} wählt das Governance Cockpit nur die \ac{lz}-Categories aus, der Rest wird abstrahiert.

\textbf{Fokus}: LZ-Kategorien mit Vereinfachung für den Endnutzer, da die Category alles umfasst, ohne dass viel ausgewählt werden muss.

\textbf{Artefakte}: versionierte \ac{lz}-Categories z.B. in Git, Governance Decisions und Evidence als Datenobjekte

\textbf{Funktionsweise}: Die Controls werden in \ac{lz}-Categories gebündelt, die alle relevanten Regeln für die verschiedenen Komponenten enthalten. Jede Category repräsentiert eine bestimmte \ac{lz}-Kategorie, die eine Risikoklasse oder einen Anwendungsfall abdeckt. Die \ac{lz}-Categories entsprechen den Regulations bzw. ihren jeweiligen Abstufungen. Beim Deployen einer \ac{lz} wählt das Governance Cockpit die entsprechende Category aus, ohne dass der Benutzer sich mit den einzelnen Controls oder Policies auseinandersetzen muss. Diese Herangehensweise bietet eine einfache Anwendung und eine klare Customer Story, da die Kategorien direkt mit den Risikoklassen verknüpft sind. Allerdings führt dies zu einer hohen Duplikation von Controls über verschiedene Kategorien hinweg, und die Rückverfolgbarkeit zu einzelnen Controls erfordert die Pflege eines Manifests, das auflistet, welche Controls in welchen Kategorien enthalten sind.

\textbf{Verantwortlichkeiten}: Der Projektverantwortliche definiert die Anforderungen für die jeweilige Foundation. Der Compliance Officer ist zuständig für die Regulations und Controls. Er prüft zudem das Mapping der \ac{lz}-Categories durch den Cloud-Experten sowie die Zuweisung der \ac{lz}-Categories zu den Foundations.

\textbf{Maintenance}: Änderungen an einer Control erfordern die Aktualisierung aller \ac{lz}-Categories, die diese Control enthalten, was den Wartungsaufwand erhöht. Es ist wichtig, einen klaren Versionierungsprozess zu etablieren, um sicherzustellen, dass Änderungen nachvollziehbar und Versionen referenzierbar sind und die Historie der Controls und \ac{lz}-Categories gepflegt wird. Dies könnte durch die Verwendung von \ac{iac} in Git erreicht werden, um die Änderungen zu verfolgen und Versionierung von Controls und Categories zu ermöglichen.

\textbf{Adaptability}: Neue Controls können durch die Erstellung und Verknüpfung neuer Versionen von \ac{lz}-Categories hinzugefügt werden. Allerdings kann dies zu einer erhöhten Komplexität führen, da die Duplikation von Controls über verschiedene Kategorien hinweg die Verwaltung und Aktualisierung der \ac{lz}-Categories erschwert.

\textbf{Abgrenzung zu anderen Varianten}: Im Vergleich zu Variante 1 und 2, welche die Controls in Bundles oder Component-zentrierte Profiles bündeln, bündelt diese Variante die Controls auf einer höheren Ebene, nämlich auf der Ebene von \ac{lz}-Kategorien.

\textbf{Tracebility-Strength}: Die Rückverfolgbarkeit zu einzelnen Controls ist in dieser Variante am schwierigsten, da die Controls in den \ac{lz}-Categories gebündelt sind und es eine hohe Duplikation von Controls über verschiedene Kategorien hinweg gibt. Es erfordert die Pflege eines Manifests, das auflistet, welche Controls in welchen \ac{lz}-Categories enthalten sind, um eine klare Rückverfolgung zu ermöglichen.

\textbf{Laufzeitkomplexität}: Deploy Foundation: O(n), Verify Compliance: O(n)

\subsection{Variante 4: LZ Provider-native Bundling}
\label{sec:trace-arch-variant4}

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{graphics/ip6-ideation-Traceability-Architecture-provider-native.drawio.pdf}
    \caption{LZ Provider Native Traceability Architektur Variante}
    \label{fig:traceability-architecture-provider-native}
\end{figure}

\textbf{Ansatz}: Controls werden direkt durch die native Policy-Abstraction des Providers dargestellt (Azure Initiatives, AWS SCP usw.). Diese werden direkt auf die Komponenten angewendet.

\textbf{Fokus}: Die Anwendung von Richtlinien auf Ressourcen, weniger auf die Nachvollziehbarkeit (à la MVP). Initiatives/Bundles existieren bereits von verschiedenen Cloud-Providern.

\textbf{Artefakte}: versionierte Richtlinien in Git, Governance Decisions und Evidence als Datenobjekte

\textbf{Funktionsweise}: In dieser Architektur-Variante werden die Controls direkt durch die nativen Richtlinienabstraktionen der Cloud-Provider dargestellt. Das bedeutet, dass die Implementierung der Controls direkt auf den Ressourcen erfolgt, ohne dass eine zusätzliche Abstraktionsschicht wie Bundles oder Profiles erforderlich ist. Diese Herangehensweise ermöglicht eine optimale Leistung und eine enge Ausrichtung an den Mechanismen der Provider, da sie die nativen Funktionen und Tools nutzt. Allerdings führt dies zu einer schlechteren Vergleichbarkeit über verschiedene Cloud-Provider hinweg, da jeder Provider unterschiedliche Formate und Strukturen für seine Richtlinien hat.

\textbf{Verantwortlichkeiten}: Der Projektverantwortliche definiert die Anforderungen für die jeweilige Foundation. Der Compliance Officer ist zuständig für die Regulations und Controls. Er prüft zudem die Implementierung der Controls durch den Cloud-Provider bzw. den Cloud-Experten, welcher direkt die nativen Richtlinien des Providers auf die Ressourcen anwendet.

\textbf{Maintenance}: Da die Controls direkt auf den Ressourcen implementiert werden, erfordert die Wartung und Aktualisierung der Controls eine sorgfältige Verwaltung der Ressourcen und ihrer zugehörigen Richtlinien. Es ist wichtig, einen klaren Versionierungsprozess zu etablieren, um sicherzustellen, dass Änderungen nachvollziehbar und Versionen referenzierbar sind und die Historie der Controls gepflegt wird. Dies könnte durch die Verwendung von \ac{iac} in Git erreicht werden, um die Änderungen zu verfolgen und Versionierung von Ressourcen und Richtlinien zu ermöglichen.

\textbf{Adaptability}: Neue Controls können durch die Erstellung und Verknüpfung neuer Ressourcen und Richtlinien hinzugefügt werden. Allerdings kann dies zu einer erhöhten Komplexität führen, da die Verwaltung der Ressourcen und ihrer zugehörigen Richtlinien über verschiedene Cloud-Provider hinweg eine Herausforderung darstellen kann.

\textbf{Abgrenzung zu anderen Varianten}: Im Vergleich zu den anderen Varianten, welche die Controls in Bundles oder Profiles bündeln, bündelt diese Variante die Controls nicht in einer zusätzlichen Abstraktionsschicht, sondern implementiert sie direkt auf den Ressourcen. Dies führt zu einer engeren Ausrichtung an den Mechanismen der Provider, aber auch zu Herausforderungen in Bezug auf Vergleichbarkeit und Rückverfolgbarkeit.

\textbf{Tracebility-Strength}: Die Rückverfolgbarkeit zu einzelnen Controls ist in dieser Variante vergleichsweise schlecht, da die Controls direkt auf den Ressourcen implementiert werden und es keine zusätzliche Abstraktionsschicht gibt, die die Controls organisiert. Die Rückverfolgbarkeit zu einzelnen Controls erfordert jedoch eine Normalisierung der Evidence über die verschiedenen Evidenceformate der Provider hinweg, um sicherzustellen, dass die Controls konsistent identifiziert und verfolgt werden können.

\textbf{Laufzeitkomplexität}: Deploy Foundation: O(n), Verify Compliance: O(1) bis O(n)
